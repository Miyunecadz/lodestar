# Lodestar — Architecture & Design

This is the finalized design for Lodestar: an AI-native workspace kit for Claude Code that coordinates multiple repositories from a single root, with knowledge that loads on demand and safety that is enforced rather than suggested.

It records not just *what* Lodestar does but *why* each choice was made, so anyone forking it understands the trade-offs before they change them.

---

## 1. Goals

1. **One launch point for many repos.** Start Claude Code at the workspace root and it understands the whole system, then works correctly inside any single repo.
2. **No `CLAUDE.md` per repo.** Centralize shared knowledge; avoid duplication and drift across repos.
3. **Load only what the task needs.** Every piece of knowledge carries an explicit *"when to load this"* trigger. Planning tasks never pull coding standards; coding tasks never pull the planning playbook.
4. **Self-extending.** Adding a repo (or a whole new workspace) is one command, not a manual doc-writing session.
5. **Portable & publishable.** The whole kit is plain files. Copy it, publish it, modify it to taste.
6. **Trustworthy, not just informed.** Anything with real cost (migrations, secrets) is *enforced*, not left to the model's discretion.

## 2. Non-goals

- Not a monorepo migration. Repos keep their own git history, CI, and independence.
- Not a new runtime or framework. Lodestar arranges *existing* Claude Code primitives (skills, hooks, agents, MCP scopes, slash commands).
- Not stack-specific. Defaults were authored on a Node/GraphQL + React + React Native workspace, but every catalog entry is stack-tagged and swappable.

---

## 3. The layered architecture

Lodestar is five independent layers. Each is optional; each does one job.

### Layer 1 — Router (`CLAUDE.md` at the workspace root)

`CLAUDE.md` loads **in full on every session**, so it must stay tiny. Lodestar's root `CLAUDE.md` is a *router*, not a knowledge base. It contains only:

- a one-line description of the workspace,
- the **repo registry** (which repos exist, one line each, pointer to `docs/repo-map.md`),
- a **loading policy**: "Do not read docs eagerly. Skills declare when they apply — trust their triggers. Do not read a repo's docs for a task in another repo."

Everything heavy lives elsewhere and is pulled in on demand. This is the single most important discipline in the whole kit: **the router must never become the knowledge base.**

### Layer 2 — Knowledge (`docs/` + on-demand skills)

Two sub-parts:

**Docs** — the heavy, hand-written reference material. Organized **project-first**, because the mental model is per-repo ("I'm working on the frontend"):

```
docs/
├── _shared/                 # belongs to NO single repo — the cross-cutting truth
│   ├── <api>-contract.md    # the spine: the contract every repo agrees on
│   ├── env-matrix.md        # environment tiers and which config maps where
│   ├── auth-model.md        # auth / permissions spanning all repos
│   ├── local-setup.md       # run the whole system locally (the runbook)
│   └── glossary.md          # domain language shared across repos
├── <repo-a>/
│   ├── architecture/        # GENERATED by Graphify (see Layer 3)
│   └── conventions.md       # hand-written, repo-specific
└── <repo-b>/ …
```

`_shared/` exists because some truth belongs to the *system*, not any repo — most importantly the API contract that links the repos at runtime. Project-first folders can't hold that; `_shared/` can.

**Skills** — the *"when to load"* layer. A skill is a folder with a `SKILL.md` whose frontmatter has a `name` and a `description`. **Only the description loads at startup** (a few hundred tokens total); the body loads only when Claude judges the task matches. That `description` field *is* the "when to load this" statement.

> A skill body should be thin and **point at the docs** (`read docs/<repo>/…`). Skills route; docs hold content. This keeps a single source of truth.

### Layer 3 — Structure (Graphify code graph)

[Graphify](https://github.com/Graphify-Labs/graphify) parses code with tree-sitter (deterministic, local, ~40 languages) and emits per repo: an interactive `graph.html`, a `GRAPH_REPORT.md`, and a queryable `graph.json` — explicitly "for querying without re-reading files." That directly serves the original goal: *refer to a map instead of re-reading source over and over.*

Graphify **replaces hand-writing** the `architecture/` docs. The `/onboard-repo` command runs it and files the outputs under `docs/<repo>/architecture/`. Because it is deterministic, re-running it is cheap — that is how Lodestar beats architecture-doc drift.

**Important caveat — the cross-repo edge.** Graphify draws *static* edges (imports, calls, inheritance). Repos that talk over an API at **runtime** (e.g. a GraphQL/REST boundary) do **not** import each other's code, so Graphify will not draw that edge. Per-repo graphs are excellent; the **cross-repo spine stays hand-written** in `docs/_shared/<api>-contract.md`. This is not a bug — it is the correct division between what a parser can know and what only a human can state.

### Layer 4 — Guardrails (enforced rules)

The pivotal distinction in Lodestar:

| | Advisory (docs / skills) | Enforced (hooks / permissions) |
|---|---|---|
| Mechanism | the AI reads it and *chooses* to comply | runs or blocks **deterministically** |
| Reliability | probabilistic | absolute |
| Use for | judgment, style, context | anything where a violation has real cost |

A doc saying "don't edit applied migrations" *will* eventually be violated — not from malice but because advisory text is probabilistic. A hook makes it impossible. Guardrails are therefore the higher-leverage half of the kit: they are what let you actually let the AI run.

**Severity policy (a deliberate default):**
- **Safety rules hard-block** — migrations, secrets, generated files. The block message *redirects* to the right action (e.g. "create a new migration with `db:new`") rather than merely failing.
- **Quality rules warn** — lint-on-edit, style — so they inform without halting flow.

Guardrails are **opt-in via a picker** (Layer's generator, §4). Auto-applying them would be presumptuous; every team wants a different subset. Most rules are emitted as declarative **hookify** files (`.claude/hookify.<name>.local.md`) because those are portable; the few needing custom logic (e.g. a per-repo lint router) are emitted as `settings.json` hooks.

### Layer 5 — Delegation (role-based agents)

Subagents run in their **own context window** and return a result — good for isolating noisy work, running a distinct tool profile, or parallel fan-out. In a multi-repo workspace the payoff is the **orchestrator pattern**: a feature that spans repos is planned at the root and dispatched to specialist workers in parallel.

**Role-based, not repo-based.** A "backend agent" has an *open-ended* scope ("do backend work") with no done-condition — that unbounded scope, not the domain knowledge, is what invites drift. A *role* ("write a migration", "review this diff") has a crisp done-condition, a minimal tool profile, and a narrow context need. So Lodestar's agents are **roles parameterized by the repo they are pointed at**, not repo owners.

Two design rules:
1. **Agents compose with skills; they never copy them.** An agent's body is thin: "you work in `<repo>`; load skill `<x>` and `docs/<y>`; here is your tool profile." Knowledge stays in the skill (single source of truth).
2. **Tool profiles are the real differentiator.** A `migration-writer` may run Bash; a `reviewer` is read-only. Getting the allowlist right matters more than the persona text.

**Where breadth lives:** in the orchestrator/main thread and the docs/graph it reads — *not* in a worker. Map at the top, hands at the bottom. One deliberately broader role, `implementer`, exists as a safety valve for cohesive multi-file features — but it is scoped to *a feature's files*, not the whole repo forever.

## 4. The generator engine (catalog + picker + manifest)

The three generator commands (`/onboard-repo`, `/guardrails`, `/gen-agents`) share one engine:

```
detect stacks present → filter the catalog to relevant entries →
present a multi-select menu (recommended pre-checked) →
write only the chosen entries → record them in the manifest
```

- **Catalog** = the reusable, publishable asset. A folder of *templates* (guardrails, agents, skills), each tagged with metadata (`stacks`, `category`, `severity`, `recommended`, …). Forking Lodestar = editing the catalog. This is a lightweight, in-repo "marketplace." The catalog is a **universal core** (`stacks: [all]`) plus **stack packs** that activate only when detected — it ships a Node·GraphQL·React·React Native pack and a Python·Django pack. See `catalog/CATALOG.md`.
- **Picker** = the command. Generic. It reads the catalog, detects stacks, and drives an `AskUserQuestion` multi-select (Claude Code's native selectable menu). Recommended entries are pre-checked; the menu only shows entries that apply to the detected stacks — so it feels curated, not a wall of toggles.
- **Manifest** = `.claude/lodestar.manifest.json`. Records the workspace, detected repos/stacks, and which guardrails/agents/skills are enabled. It is your **lockfile**: commit or share it, and `re-apply` reproduces the exact setup elsewhere.

### Stack detection (heuristics)

`/onboard-repo` classifies a repo by cheap signals, e.g.:

| Signal | Stack tag |
|---|---|
| `dbmate.yml` present | `node-dbmate` |
| `apollo-server-*` in deps | `graphql-apollo-server` |
| `@apollo/client` in deps | `graphql-apollo-client` |
| `@craco/craco` in deps | `react-craco` |
| `react-native` in deps | `react-native` |
| `.husky/` present | `has-husky` |
| eslint config / dep present | `has-eslint` |
| `bull` / `ioredis` in deps | `redis-queue` |

Catalog entries list the stacks they apply to; the picker intersects detected stacks with each entry's `stacks` (or `[all]`).

## 5. MCP: per-workspace tool access

Per-folder MCP configuration is **native to Claude Code** — Lodestar does not build it, it documents it. Three scopes:

| Scope | Location | Behavior |
|---|---|---|
| local | `~/.claude.json`, keyed by folder path | private to you, that folder only |
| **project** | **`.mcp.json` at the folder root** | the folder's own server set; committable/shareable |
| user | `~/.claude.json` global | everywhere |

Precedence: **local > project > user > plugin > claude.ai connectors.** Launch from workspace A → its servers; from workspace B → different servers. The clean pattern: put the **server list** in `.mcp.json` (safe to share — no secrets), each person supplies their **own token** via local scope.

Lodestar ships **template `.mcp.json` files** per workspace type (e.g. an Atlassian/Rovo set for Jira+Bitbucket; a GitHub+Asana set) plus a one-line "run `/mcp` to authenticate" note. A full MCP *picker* is deliberately **out of scope for v0.1**: it would mostly duplicate the built-in `/mcp` and `claude mcp add`, and no command can complete an interactive OAuth for you. Config can be automated; credentials cannot.

## 6. Placement: where the Lodestar files live in a target workspace

The workspace root is often **not itself a git repo** (just a folder holding sibling repos). Two options, chosen per workspace:

- **Personal / untracked (default).** Lodestar files sit at the root as plain files. Fastest, zero risk to the repos, nothing to commit. Not shared or backed up until you decide to.
- **Git workspace repo.** `git init` the root and `.gitignore` the sub-repos. Version-controlled and directly shareable — the "virtual monorepo" pattern.

Lodestar itself (this repo) is always a normal git repo — it is the kit. The choice above is about the *target* workspace you install it into.

## 7. Trade-offs (the honest list)

**Pros**
- Token-efficient: a ~500-token index at startup instead of tens of thousands of always-loaded docs.
- Cross-repo awareness from a single launch point.
- Enforced safety (not hopeful advice) for high-cost actions.
- Self-extending and fully portable — the catalog is the product.

**Cons / risks**
- **Skill/agent discovery depends on `description` quality.** Vague triggers load the wrong thing or nothing. This is the real ongoing maintenance cost.
- **Docs drift.** Graphify fights it for *structure*; hand-written `_shared/` docs still need refreshing. Generated ≠ maintained.
- **Complexity budget.** Five layers is a lot. Adopt them in order; add agents last, and only the ones you will actually delegate to.
- **Portability limits.** Declarative hookify rules travel well; any rule needing a shell script (the lint router) is the least portable part — keep those clearly marked.
- **Generator quality.** Auto-generated "best practices" tend to be generic. The high-value docs (architecture rationale, gotchas, why-not-X) still need a human. Treat generators as bootstrap + refresh, then curate.

## 8. Decision log

Choices made during design, with the reasoning, so forks can revisit them deliberately.

| # | Decision | Rationale |
|---|---|---|
| D1 | Centralize at the root; **no per-repo `CLAUDE.md`** | avoid duplication/drift; enable a cross-repo view |
| D2 | Root `CLAUDE.md` is a **thin router**, knowledge in skills | `CLAUDE.md` loads every session; skills load on demand |
| D3 | Docs organized **project-first**, plus `_shared/` | mirrors the per-repo mental model; `_shared/` holds system-level truth |
| D4 | **Graphify** generates architecture; contract stays hand-written | parser knows static edges; humans state runtime/semantic ones |
| D5 | Guardrails are **opt-in via a picker** | every team wants a different subset; auto-applying is presumptuous |
| D6 | **Hard-block safety, warn on quality** | match enforcement strength to the cost of a violation |
| D7 | Agents are **role-based, not repo-based** | narrow *task* scope (not narrow domain) is what prevents drift |
| D8 | Agents **reference** skills, never copy them | single source of truth; copies drift |
| D9 | Keep one **task-scoped `implementer`** role | safety valve for cohesive multi-file features |
| D10 | One shared **catalog + picker + manifest** engine for all generators | less to build; consistent UX; the catalog is the shareable product |
| D11 | MCP isolation documented, **not built**; ship templates only | it is native; a picker would duplicate built-ins and can't do OAuth |

## 9. Roadmap

- **v0.1 (initial, unreleased)** — router + four generator commands; catalog with a universal core plus Node·GraphQL·RN and Python·Django packs; core-vs-packs signposting (`catalog/CATALOG.md`); doc & MCP templates; install script.
- **Later** — a skills picker (same engine); a `re-apply <manifest>` command; a lint-router settings hook generator; optional MCP picker; CI recipe to refresh Graphify graphs on a schedule.

See [`CONCEPTS.md`](CONCEPTS.md) for the mental models and [`EXTENDING.md`](EXTENDING.md) to add your own catalog entries.
